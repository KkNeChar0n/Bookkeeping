## 需求
### 需求:调账为两卡同变的零和交易
系统必须提供"调账"操作。用户在当前卡上选择调出或调入，并指定对手卡。调出必须从当前卡减余额、对手卡加余额；调入必须从当前卡加余额、对手卡减余额。一次调账对系统总余额的净影响必须为零。

#### 场景:调出到另一张卡
- **当** 用户在卡 A 上选择"调出"金额 N 到卡 B
- **那么** 系统必须使卡 A 余额减少 N、卡 B 余额增加 N，且全系统总余额不变

#### 场景:从另一张卡调入
- **当** 用户在卡 A 上选择"调入"金额 N（来自卡 B）
- **那么** 系统必须使卡 A 余额增加 N、卡 B 余额减少 N，且全系统总余额不变

#### 场景:必须指定对手卡
- **当** 用户选择调账但未指定对手卡
- **那么** 系统必须拒绝提交并提示选择对手卡

### 需求:调账原子写入
系统必须把一次调账作为单个原子事务写入两卡的变动，禁止出现只改了一张卡的半截状态。

#### 场景:写入失败整体回滚
- **当** 调账写入过程中任一卡的更新失败
- **那么** 系统必须回滚整次调账，使两卡都不发生变化

### 需求:调账排除于收支统计
系统必须把 type=transfer 的记录排除在收入与支出统计之外。

#### 场景:统计排除调账
- **当** 系统计算某期收入或支出
- **那么** 系统必须排除全部调账记录，只统计真实收入与支出

### 需求:预算层与实际层通用
系统必须在预收预算与当日记账两个层面都支持相同语义的调账。

#### 场景:预算层调账
- **当** 用户在预算快照中对某卡执行调账
- **那么** 系统必须按零和规则调整两卡在该快照的派生余额

#### 场景:实际层调账
- **当** 用户在当日记账中执行调账
- **那么** 系统必须按零和规则调整两卡的实际派生余额
