## 需求
### 需求:当日记账录入收入或支出
系统必须允许用户在当日记账界面先选择一张卡（默认选中默认卡），再选择收入或支出，并选择一个分类，录入金额、日期与备注。

#### 场景:记一笔支出
- **当** 用户选定某卡、选择"支出"、填写金额并选择支出分类后提交
- **那么** 系统必须以 type=out 持久化该笔流水，并据此下调该卡实际余额

#### 场景:记一笔收入
- **当** 用户选定某卡、选择"收入"、填写金额并选择收入分类后提交
- **那么** 系统必须以 type=in 持久化该笔流水，并据此上调该卡实际余额

#### 场景:默认选中默认卡
- **当** 用户进入当日记账界面且尚未手动选卡
- **那么** 系统必须默认选中用户设定的默认卡

### 需求:分类可选
系统必须为收入与支出分别提供可选分类清单（如收入：工资/报销/利息/投资收益/其他；支出：餐饮/房租/交通/购物/医疗/其他），且分类清单必须可扩展。

#### 场景:选择分类
- **当** 用户为一笔流水选择分类
- **那么** 系统必须把该分类随流水一并持久化，供统计使用

### 需求:实际层余额可手改并自动留痕
系统必须允许用户在实际层手动修改某卡的当前余额（用于吸收银行利息等）。手改余额时，系统禁止直接覆盖余额字段，必须自动生成一条"调整"(type=adjust)流水，其金额等于"目标余额 − 当前派生余额"，使余额仍由流水派生且历史可追溯。

#### 场景:手改余额生成调整流水
- **当** 用户把某卡实际余额从 X 改为 Y
- **那么** 系统必须生成一条金额为 (Y − X) 的 adjust 流水，使派生余额变为 Y，并保留该调整记录

#### 场景:调整流水不计入收支统计
- **当** 系统进行收入/支出统计
- **那么** 系统必须排除 type=adjust 的流水，使利息/校正不污染消费或收入统计
