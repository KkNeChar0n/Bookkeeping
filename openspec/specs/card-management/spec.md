## 需求
### 需求:用户可自定义管理卡片
系统必须允许用户创建、重命名、删除卡片，并为每张卡设置初始余额。卡片名称必须由用户自定义，系统禁止写死 ZS/YZ 等固定命名。

#### 场景:创建卡片
- **当** 用户提交一张新卡片，填写名称与初始余额
- **那么** 系统必须持久化该卡片，并以初始余额作为其余额计算的起点

#### 场景:重命名卡片
- **当** 用户修改某卡片名称为新的非空名称
- **那么** 系统必须更新该名称，且不影响该卡已有的余额与流水

#### 场景:删除卡片
- **当** 用户删除一张卡片
- **那么** 系统必须移除该卡片；若该卡仍存在关联流水或被设为默认卡，系统必须提示并阻止删除或要求先处理关联数据

### 需求:默认卡设置
系统必须允许用户将任意一张卡片标记为默认卡，且同一时刻最多只有一张默认卡。

#### 场景:设置默认卡
- **当** 用户将某卡片设为默认卡
- **那么** 系统必须把该卡标为默认，并清除其他卡片的默认标记

#### 场景:当日记账使用默认卡
- **当** 用户进入当日记账界面
- **那么** 系统必须默认选中默认卡作为当前记账卡

### 需求:卡片排序
系统必须支持用户自定义卡片展示顺序。

#### 场景:调整顺序
- **当** 用户调整卡片排序
- **那么** 系统必须按新顺序持久化，并在预算、记账、卡片展示等处按该顺序呈现

### 需求:非银行卡平等参与调账
系统必须把所有卡片（包括基金等非银行卡，如 JJ）一视同仁，使其与银行卡平等地参与调账与余额计算。

#### 场景:非银行卡参与调账
- **当** 用户对一张非银行卡执行调出或调入
- **那么** 系统必须像处理普通卡一样调整其余额，不得因卡类型而拒绝
