## 新增需求

### 需求:后端持久化数据
系统必须将卡片、预算快照、预算明细与流水持久化到后端数据库(Postgres)，使数据在客户端重启或更换设备后仍然存在。

#### 场景:数据持久化
- **当** 用户在一台设备上录入数据后关闭客户端
- **那么** 系统必须在后端持久化这些数据，重新打开时数据完整保留

### 需求:多设备同步
系统必须以服务端为单一数据来源，使同一用户在不同设备访问时看到一致的数据。

#### 场景:跨设备一致
- **当** 用户在设备 A 录入一笔流水，随后在设备 B 打开应用
- **那么** 设备 B 必须能读取到设备 A 录入的最新数据

### 需求:单人鉴权
系统必须对所有数据读写接口要求鉴权（单人密码换取访问令牌），未鉴权请求必须被拒绝。

#### 场景:未鉴权访问被拒
- **当** 一个未携带有效令牌的请求访问数据接口
- **那么** 系统必须拒绝该请求并返回未授权错误

#### 场景:鉴权后访问
- **当** 用户以正确密码登录获取令牌并携带其访问接口
- **那么** 系统必须允许该请求读写其数据

### 需求:可扩展的交易数据模型
系统必须以单张流水表承载收入、支出、调账、调整四类记录，用类型字段区分，调账用对手卡字段关联，使新增交易类型无需更改表结构。

#### 场景:新增类型不破坏结构
- **当** 需要支持一种新的交易语义
- **那么** 系统必须能通过新增类型值与既有字段表达，而无需对核心流水表做破坏性结构变更
