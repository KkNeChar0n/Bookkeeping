## 上下文

用户当前用一张 Excel 记账，结构分两块：第 1~8 行是"预收预算"（按日期 41/45/410/426 拍快照，逐卡填收/支，余额滚动累加），第 12 行向下是"实际收支"。卡片有 ZS/YZ/GS/JS/JJ 五张，分别承担收款存储/存储/投资/消费/基金等用途，每月 1 号从 ZS 向其他卡调账。痛点：无法多设备同步、人工算余额易错、超支不直观、调账与真实收支混在一起。

约束（用户已拍板）：
- 多设备同步，自建后端数据库（路线 B：React 前端 + Node API + Postgres）。
- 移动端优先自适应。
- 不做巨石项目，也不过度设计，保证延展性（单人使用，简单鉴权即可）。

## 目标 / 非目标

**目标：**
- 把"预收预算"与"当日记账"两块结构化，余额在预算层自动算、在实际层可手改。
- 调账作为零和交易，两卡同变，永不进收支统计。
- 当日记账顶部展示对应预算快照的各卡卡片，实际余额 < 预算余额时标红超支。
- 后端持久化 + 多设备同步；前端移动优先。
- 卡片、分类可自定义扩展，新增交易类型不改表结构。

**非目标：**
- 多用户/团队协作、权限体系（单人使用，简单鉴权）。
- 自动从银行/账单导入；复杂记账（多币种、汇率、发票）。
- 复杂报表 BI；当前总结仅做预算 vs 实际与 y2y/m2m。
- 离线优先/本地冲突合并（同步以服务端为准即可）。

## 决策

**D1 — 单表交易模型。** `transactions` 一张表，用 `type ∈ {in, out, transfer, adjust}` 区分；调账用 `peer_card_id` 关联对手卡，余额手改自动生成 `adjust` 记录。
理由：新增类型不动表结构=延展性；统计时按 `type` 过滤（排除 transfer/adjust）即可。
替代方案：每类型一张表 → 拆分过细、跨表聚合麻烦，弃。

**D2 — 余额=派生 vs 持久化。** 预算层余额完全派生（初始余额 + Σ收 − Σ支 ± 调账），不存"余额"字段，避免对不上。实际层允许手改：手改时不直接写余额，而是生成一条 `adjust` 流水（金额=目标余额−当前推算余额），余额仍由流水派生。
理由：留痕、单一事实来源、永远连得上账。
替代方案：余额作为可写字段 → 与流水不一致风险，弃。

**D3 — 预算覆盖周期映射。** 对某天 D，取"日期 ≥ D 的最近一个预算快照"作为对比基准；若 D 超过所有快照日期，则取最后一个快照并附"该期预算未填"提示。
理由：符合用户"6.1~6.5 都对比 6.5 预算"的语义（快照管它日期之前那段周期）。

**D4 — 超支判定。** 超支 = 实际余额 < 该期预算余额（按卡）。顶部卡片逐卡展示：预算余额、实际余额、差额，超支整张标红。
理由：直观、与 Excel 的余额思路一致（已与用户确认比余额而非比支出）。

**D5 — 调账原子性。** 一次调账在 API 层作为单个事务写入：当前卡一条出向、对手卡一条入向（或单条记录 + peer 关联，二选一实现），保证两卡同变、零和。统计聚合一律排除 `type=transfer`。
理由：避免半截调账导致总额漂移。

**D6 — 技术栈。** 前端 React + Vite + TypeScript，移动优先自适应（CSS 响应式，轻量状态管理，不引入重型框架）；后端 Node + Fastify，分层 routes→service→repo；ORM 用 Prisma（迁移 + 类型）；数据库 Postgres；鉴权用单人密码换 token（如 JWT），所有 API 需鉴权。
理由：轻量、类型安全、延展，符合"不巨石不过度"。

**D7 — 仓库结构。** `apps/web`、`apps/api`、`infra`（Prisma schema/迁移）。前后端分离但同仓，便于单人维护。

## 风险 / 权衡

- [余额派生在数据量大时聚合慢] → 单人记账数据量小；必要时按卡缓存"截至某快照的余额"做物化，不影响事实来源。
- [手改余额语义被误用为记一笔收入] → 明确 `adjust` 不进收支统计，仅作余额校正与利息留痕；UI 文案区分"改余额（利息/校正）"与"记一笔收入"。
- [调账非原子写入导致总额漂移] → API 层事务包裹两卡变动（D5）。
- [覆盖周期边界（超过所有快照）易困惑] → 明确提示"该期预算未填"，不静默显示旧值（D3）。
- [自建后端需部署运维] → 单人单实例，Postgres + 单进程 Node 足够；提供最简部署说明。
- [简单鉴权安全性] → 单人自用、HTTPS + 强密码 + token 过期即可，不做多因子。

## 迁移计划

- 全新项目，无存量系统需迁移。
- 历史 Excel 数据可后续手工补录或一次性导入（非本期范围，Open Question）。
- 回滚：单人工具，部署失败回退到 Excel 即可；数据库迁移用 Prisma migrate，可回退。
