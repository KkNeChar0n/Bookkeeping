## 为什么

目前用 Excel 记账：第 1~8 行手工维护"预收预算"快照、第 12 行向下记录实际收支，靠人工滚动余额、人工核对预算与实际。这种方式无法多设备同步、容易算错余额、超支不直观、调账（卡间换口袋）和真实收支混在一起污染统计。需要一个移动端友好、带后端数据库可多设备同步的小工具，把"预收预算"与"当日记账"两块结构化，并自动做预算 vs 实际的超支提示。

## 变更内容

- 新增**卡片管理**：用户自定义卡片（名称、初始余额、默认卡、排序），不写死 ZS/YZ 等命名；JJ 等"非银行"卡与银行卡平等参与调账。
- 新增**预收预算**：按日期拍快照，每张卡填收/支/调账，余额自动滚动计算。
- 新增**当日记账**：选卡（含默认卡）→ 选收入/支出 → 选类型；实际层余额允许手动修改（吸收利息等），手改自动补一条"调整"流水留痕。
- 新增**调账**：一笔记录、两卡同时变动、零和；在统计中永不计入收入/支出。
- 新增**预算 vs 实际对比**：当天记账顶部展示该期预算快照的各卡卡片；当实际余额 < 该期预算余额时标红超支。对比基准 = 日期 ≥ 当天的最近一个预算快照。
- 新增**总结**：预算 vs 实际汇总，以及年同比(y2y)/月环比(m2m)。
- 新增**后端与多设备同步**：自建 React 前端（移动优先自适应）+ Node API + Postgres，单人简单鉴权，数据持久化并多设备同步。

## 功能 (Capabilities)

### 新增功能
- `card-management`: 自定义卡片的增删改、初始余额、默认卡、排序；非银行卡平等参与调账。
- `budget-planning`: 按日期的预算快照，逐卡收/支录入，余额自动滚动计算。
- `daily-bookkeeping`: 当日选卡记收入/支出/分类，实际层余额可手改并自动补"调整"留痕。
- `account-transfer`: 调账模型——调出/调入，两卡同变、零和、不进收支统计；预算层与实际层通用。
- `budget-actual-comparison`: 覆盖周期映射、顶部各卡对比卡片、实际余额 < 预算余额的超支标红。
- `summary-reporting`: 预算 vs 实际汇总与年同比/月环比统计（调账排除在外）。
- `sync-backend`: 后端 REST API、Postgres 持久化、单人鉴权、多设备同步与数据契约。

## 影响

- 全新代码库（此前仅有 Excel）。新建 `apps/web`（React + Vite + TS，移动优先）、`apps/api`（Node + Fastify，REST，分层 routes→service→repo）、`infra`（Postgres + Prisma 迁移）。
- 核心数据表：`cards` / `budget_snapshots` / `budget_lines` / `transactions`（单表用 `type` 区分 in|out|transfer|adjust，转账用 `peer_card_id` 关联）。
- 需要部署环境：一台可运行 Node + Postgres 的服务器及对外访问（多设备同步前提）。
- 依赖新增：React/Vite、Fastify、Prisma、Postgres 驱动；前端轻量状态方案（不引入重型框架）。
